    1. Create src/pipeline/signal-detector.ts:
       - Import bus, Signal type, logger, bs58
       - Constants for PumpFun:
         PUMP_PROGRAM = '6EF8rrecthR5Dkzon8Nwu78hRvfCKubJ14M5uBEwF6P'
         PUMP_BUY_DISCRIMINATOR = Buffer.from('66063d1201daebea', 'hex')
         PUMP_SELL_DISCRIMINATOR = Buffer.from('33e685a4017f83ad', 'hex')
         PumpFun account indices: MINT=2, USER=6
       - Constants for PumpSwap (post-migration buys per locked decision TRADE-05):
         PUMPSWAP_PROGRAM = 'PSwapMdSai8tjrEXcxFeQth87xC4rRsa4VA5mhGhXkP'
         PUMPSWAP_BUY_DISCRIMINATOR = derive using same Anchor pattern: first 8 bytes of SHA-256("global:buy")
         NOTE: The PumpSwap buy discriminator MUST be verified at runtime or during development.
         Look up the PumpSwap IDL or use the pumpfun-pumpswap-sdk repo (https://github.com/Erbsensuppee/pumpfun-pumpswap-sdk) to confirm the exact discriminator bytes and account indices for PumpSwap buy instructions.
         PumpSwap account indices for mint and user will likely differ from PumpFun -- check the IDL.
         If the discriminator or indices cannot be confirmed, add a TODO comment and log a warning at startup: "PumpSwap discriminators unverified -- PumpSwap signals may be missed"
       - Export function initSignalDetector(): void
         - Subscribe to bus.on('webhook:received', handler)
         - Handler decodes each instruction in the transaction:
           a. Get programId from accountKeys[ix.programIdIndex]
           b. Decode instruction data from base58 using bs58
           c. If programId === PUMP_PROGRAM:
              - Compare first 8 bytes against PUMP_BUY/SELL discriminators
              - Extract mint and user from accountKeys using PumpFun account indices
              - Set signal.source = 'pumpfun'
           d. If programId === PUMPSWAP_PROGRAM:
              - Compare first 8 bytes against PUMPSWAP_BUY_DISCRIMINATOR
              - Extract mint and user from accountKeys using PumpSwap account indices
              - Set signal.source = 'pumpswap'
              - NOTE: Only BUY detection needed for PumpSwap (we copy buys, not sells on PumpSwap)
           e. If valid buy/sell detected, emit bus.emit('signal:detected', signal)
           f. If no matching instruction found, silently skip (not all txs are PumpFun/PumpSwap)
         - Wrap in try/catch -- log errors but never crash. Malformed webhooks should be logged and skipped.
         - Log each detected signal at info level: source (pumpfun/pumpswap), type, mint (first 8 chars), user (first 8 chars), signature (first 12 chars)

    2. Create src/pipeline/rug-filter.ts:
       - Import bus, Signal, ValidatedSignal, logger, Connection from @solana/web3.js, config
       - Export function initRugFilter(connection: Connection): void
         - Subscribe to bus.on('signal:detected', handler)
         - Handler (async):
           a. Only process BUY signals (SELL signals should be emitted directly as they bypass rug check -- emit signal:detected sells forward differently, see note below)
           b. For BUY: call connection.getTokenLargestAccounts(new PublicKey(signal.mint)) and connection.getTokenSupply(new PublicKey(signal.mint)) in parallel via Promise.all
           c. Calculate each holder's percentage of total supply
           d. If ANY holder has >5%: emit bus.emit('signal:rejected', signal, reason) with reason like "Wallet holds X.X% of supply (limit: 5%)"
              // This single >5% check covers BOTH the dev wallet check and the single wallet cap
              // from requirements (dev wallet <=5% AND no individual wallet >5%). Since we check
              // ALL top-20 holders against the same 5% threshold, any dev wallet or non-dev wallet
              // exceeding the limit is caught. Functionally equivalent to two separate checks.
           e. If all holders <=5%: emit bus.emit('signal:validated', { ...signal, rugCheckPassed: true })
           f. Log result at info level: passed/rejected with reason
         - Handle RPC errors gracefully -- if getTokenLargestAccounts fails, reject the signal with reason "RPC error during rug check" (fail closed, not open)

       NOTE on SELL signals: SELL signals don't go through rug filter. They represent tracked wallets selling, which triggers copy-sell exit logic. The signal detector should still emit them as signal:detected, but the rug filter should forward SELL signals directly by emitting a separate event or by having the position manager listen to signal:detected for SELL type. Cleanest approach: rug filter ignores SELL signals entirely, and position manager subscribes to signal:detected directly for SELL-type signals.
